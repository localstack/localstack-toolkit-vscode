## Context

The resource browser resolves a `ServiceProvider` per AWS service through `ProviderFactory`, which holds a fixed map of seven hand-written providers. Each provider hand-codes a `resourceTypes` map (id → `[singular, plural]`), `getResourceArns` (SDK `List*`), `describeResource` (SDK `Describe*` → a curated field list), and `getArnResourceNameForCloudFormationResource` (CFN type → ARN). Consumers are the Resources view (drill-down + ARN leaves), Resource Details (`describeResource`), `cfnStackModel` (CFN → focus), and `metamodelFocus` ("All Resources").

LocalStack's published coverage data lives at `localstack-docs/src/data/coverage/*.json` — **117 services**, each `{ service, pro_support, operations: [ { OpName: { implemented, availability, ... } } ] }`. This is the authoritative, static service set. It enumerates **API operations**, not resource types.

The goal: cover the full published service set, with **every** service a real curated provider (resource types, live listing, per-service detail fields), authored declaratively to be tractable, with no runtime discovery and no generic fallback. The browser targets both the LocalStack emulator and real AWS cloud profiles.

## Goals / Non-Goals

**Goals:**
- A static, coverage-derived **service manifest** as the single source of truth for which services exist.
- Every manifest service is a **curated provider**: declares its resource types, lists live resources, and describes a resource with **per-service fields**.
- A **declarative provider format + engine** that makes ~117 services feasible without sacrificing functionality; imperative `ServiceProvider` remains an escape hatch.
- **Completeness** enforced by a test (every manifest service has a provider), delivered in **batches**.
- No runtime service discovery (`/_localstack/health`, Cloud Control) and no generic provider tier.

**Non-Goals:**
- Parent/child **nesting** of resources. The tree stays flat; cross-resource **hyperlinks** are deferred to future work.
- Displaying **availability** (community/pro). Because the browser also targets real AWS, every service is treated as fully available to everyone; the distinction is neither stored nor shown.
- Interactive detail behaviors (links, open-in-editor) — already deferred by the `resource-browser` spec.
- Landing all 117 providers in this change. This change delivers the manifest, engine, parity migration of the 7, and Batch 1; later batches finish coverage.
- Hand-authoring every detail field up front (see D8: fields are generated by importance, then refined).
- Multi-account support beyond the current single-account assumption.

## Decisions

### D1: Static manifest generated from LocalStack coverage data

Add `resources/service-manifest.json` (bundled) + a loader under `src/platforms/aws/services/`. A checked-in generator reads `localstack-docs/src/data/coverage/*.json` and emits one entry per service: `{ id, name }`, where `id` is **AWS's own service code** (the SDK/`endpoint` id, e.g. `s3`, `secretsmanager`, `cognito-idp`, `logs`, `events`). **Every** service is included regardless of community/pro availability, and availability is neither recorded nor displayed. The generated file is committed so the build needs no network.

- **Regeneration is on-demand only**: a developer runs the generator when someone notices the manifest is out of date. No CI job or scheduled refresh; staleness is acceptable between manual regenerations.
- *Why static*: the published list is the contract; an instance's `/_localstack/health` reports runtime availability, not the supported set, and doesn't exist for cloud profiles.
- *Alternatives rejected*: `/_localstack/health` discovery, Cloud Control, scraping the pricing SPA — all rejected per the requirement that supported-set discovery never hits a running service.

### D2: Every service is curated — no generic tier

`ProviderFactory` registers a curated provider for each manifest service; `getProviderForService(id)` returns it. There is no generic/Cloud-Control fallback: a service with no provider yet simply does not appear (transitional, during rollout — a missing service is just absent, with no "coming soon" affordance). A **completeness test** asserts every manifest service id has a registered provider; while batches are in flight it runs as a coverage tracker and must be green when the change is "done."

### D3: Declarative provider definitions executed by a shared engine

A service is authored as data, with the detail fields expressed as a **data spec** (not a closure) so the generator (D8) can emit them and humans can edit them:

```
defineService("s3", "S3", {
  bucket: {
    singular: "Bucket", plural: "Buckets",
    list:   c => c.listBuckets().Buckets,   id: b => b.Name,
    cfn:    "AWS::S3::Bucket",
    detailFrom: "self",                       // or a Describe op name
    detail: [ { label: "Name", path: "Name", type: NAME },
              { label: "Created", path: "CreationDate", type: DATE }, ... ],
  },
})
```

A single engine adapts a definition to the `ServiceProvider` interface (resource types, `getResourceArns`, `describeResource` by walking each field's `path` over the response, CFN mapping). The imperative `ServiceProvider` class stays as the **escape hatch** for services whose listing/detail can't be expressed declaratively. This is *not* a generic provider — nothing works without a per-service definition; it is curation expressed as data.

- **Parity requirement**: the existing 7 providers are re-expressed as definitions and their output diffed against today's behavior to prove "same functionality."

### D4: Resource types are an editorial curation layer

Coverage data has operations, not resource types. Each service definition **declares** its resource types. CloudFormation resource-type names (e.g. `AWS::S3::Bucket`) are taken from existing knowledge and committed once — **not** verified against live CloudFormation data, since they rarely change. Resource-type sets are bounded to listable, browsable types (≤5 per service in early batches).

### D5: Flat resource structure; hyperlinks deferred

The tree stays `profile → region → service+type → resource`. Naturally hierarchical types (API Gateway Stages, Cognito clients, Log Streams) are listed as flat, region-wide rows for now. Navigation between related resources will later be done via **hyperlinks** in Resource Details (the webview already supports `command:` URIs), not tree nesting. So a definition's `list` takes only profile/region — no parent-id input — in this change.

### D6: Listing/detail via per-service SDK calls, honoring `endpoint_url`

Live listing and detail use the service's AWS SDK `List*`/`Describe*` calls through `AWSConfig.getClientConfig` (so the same definitions work against LocalStack and real AWS). Identifiers are the resource ARN where the API returns one, else the resource's primary id; the Resources view leaf tolerates a non-ARN identifier (shows it verbatim).

### D7: Batched delivery; Batch 1 defined

1. Build the declarative engine; migrate the 7 existing providers to definitions (parity).
2. Land the manifest + loader + completeness test (as a coverage tracker).
3. **Batch 1** — 10 popular, currently-uncovered services, with these resource types (flat):

   | Service (`id`) | Resource types (CFN) |
   |---|---|
   | S3 (`s3`) | `Bucket` |
   | API Gateway (`apigateway`) | `RestApi`, `Stage`, `ApiKey`, `UsagePlan`, `Authorizer` |
   | SSM (`ssm`) | `Parameter`, `Document`, `MaintenanceWindow`, `Association`, `PatchBaseline` |
   | Secrets Manager (`secretsmanager`) | `Secret` |
   | Kinesis (`kinesis`) | `Stream`, `StreamConsumer` |
   | CloudWatch Logs (`logs`) | `LogGroup`, `LogStream`, `MetricFilter`, `SubscriptionFilter`, `Destination` |
   | EventBridge (`events`) | `EventBus`, `Rule`, `ApiDestination`, `Connection`, `Archive` |
   | KMS (`kms`) | `Key`, `Alias` |
   | Cognito (`cognito-idp`) | `UserPool`, `UserPoolClient`, `IdentityPool`, `UserPoolGroup` |
   | ECR (`ecr`) | `Repository` |

4. Subsequent batches until the completeness test is green.

### D8: Detail fields are auto-generated by importance, then refined

Detail-field selection is a **build-time authoring aid**, not runtime behavior. A dev-time generator reads each resource type's Describe/Get (or list-item) output shape from **offline AWS API models** (the `aws-sdk` v2 `apis/*.normal.json` models, or equivalently botocore `service-2.json` — used only by the generator, never bundled at runtime), ranks the members by an importance heuristic, maps each to a `FieldType`, and emits the `detail` data spec (D3) into the committed definition. Developers then hand-refine based on feedback.

- **Importance heuristic** (descending): identifiers/names (`Name`, `*Name`, `Id`, `*Id`, `Arn`, `*Arn`) → status/state → type/mode → timestamps → other top-level scalars. Nested objects/arrays are collapsed to `JSON` or dropped beyond depth 1; `ResponseMetadata`, pagination tokens, and large blobs are excluded. Capped at ~12 fields. Type map: `timestamp→DATE`, integer/float→`NUMBER`, `string→NAME` (→`ARN` if the name ends with `Arn`, →`LOG_GROUP` if it looks like a log group), structure/list/map→`JSON`.
- **Runtime still renders a fixed, selected, typed subset** — the committed spec — not the raw unfiltered response. So this is not a generic runtime dump; it is a generated-then-curated per-service list.
- **Known limits (accepted)**: services whose detail spans multiple calls (e.g. S3 bucket: `GetBucketVersioning`/`GetBucketLocation`/…) or whose only data is the list item get a partial first cut from a single shape and are hand-finished; "refine later" covers this.

## Risks / Trade-offs

- **Scale & ongoing maintenance (117 services × fields)** → Declarative definitions + the D8 generator minimize hand-work; detail fields are auto-seeded and refined on feedback. Batched delivery keeps each increment reviewable; the completeness test prevents silent gaps.
- **Auto-selected fields may miss or mis-rank what matters** → Accepted; fields are committed and editable, refined from user feedback. Multi-call/list-only services need hand-finishing.
- **Declarative engine expressiveness** → Some services won't fit; the imperative `ServiceProvider` escape hatch covers them.
- **Parity regressions when migrating the 7** → Diff declarative output against current providers in tests before removing the imperative versions.
- **Non-ARN identifiers (flat listing)** → Leaf rendering tolerates opaque identifiers; ARNs used where available.
- **Manifest drift from LocalStack's real matrix** → Generated + committed from `localstack-docs`, regenerated on demand when noticed; a test that curated services match manifest ids guards rot.

## Migration Plan

Engine → migrate 7 (parity, diff-tested) → manifest + loader + completeness test → field generator → Batch 1 (10 services above) → later batches. Rollback is additive behind `ProviderFactory`: reverting to the fixed curated map restores current behavior with no data migration.

## Open Questions

All prior open questions are resolved: no live-CFN verification (D4); on-demand manifest regeneration (D1); availability not stored or displayed (D1, Non-Goals); detail fields auto-generated by importance then refined (D8); not-yet-curated services are simply absent (D2). Remaining tuning details (exact field cap, which offline model source to vendor for the generator) are implementation choices to settle during the engine/generator tasks, not blockers.
